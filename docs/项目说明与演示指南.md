# 项目说明与演示指南

## 一、项目是做什么的？

这是 Stevens CS565（软件架构与组件化设计）课程项目。我们使用 FTDI FT245R (UM245R) USB 芯片，通过 USB 控制 8 个 GPIO 引脚，完成了**两个阶段**的开发：

### 阶段一：ioLibrary 静态库（已完成）

将原始的单文件 C 代码重构为 **C++ 可复用静态库**，按照 UML 类图设计了 4 个类：

| 类 | 职责 |
|---|------|
| **FtdiDevice** | 封装所有 FTDI 硬件调用（open/close/read/write），所有 `FT_*` 函数只在这个类里 |
| **ioBuffer** | 管理堆内存缓冲区（malloc/free） |
| **ioWrite** | 按指定频率向设备写入数据（单线程阻塞循环） |
| **ioRead** | 按指定频率从设备读取数据（单线程阻塞循环） |

**演示程序**：`blink_test`（main.cpp）— LED 以 1Hz 和 2Hz 频率闪烁。

### 阶段二：多线程数据采集管道（已完成）

在 ioLibrary 基础上扩展了 **3 个新类**，实现生产者-消费者多线程管道：

| 类 | 职责 |
|---|------|
| **CircularBuffer** | 线程安全的环形缓冲区，使用 `std::mutex` + `std::condition_variable` 实现阻塞同步 |
| **ThreadedReader** | 在独立线程中从 FTDI 设备读取数据，写入 CircularBuffer（生产者） |
| **ThreadedWriter** | 在独立线程中从 CircularBuffer 读取数据，写入文件或另一个 FTDI 设备（消费者） |

**演示程序**：`pipeline`（pipeline.cpp）— 多线程数据采集，支持输出到文件或 FTDI 设备。

## 二、我们实现了什么？

### 完整的文件清单

```
ioLibrary/                         ← 静态库 libioLibrary.a
├── FtdiDevice.h / .cpp            ← 阶段一：设备封装
├── ioBuffer.h / .cpp              ← 阶段一：缓冲区管理
├── ioRead.h / .cpp                ← 阶段一：单线程读
├── ioWrite.h / .cpp               ← 阶段一：单线程写
├── CircularBuffer.h / .cpp        ← 阶段二：线程安全环形缓冲区
├── ThreadedReader.h / .cpp        ← 阶段二：多线程读
└── ThreadedWriter.h / .cpp        ← 阶段二：多线程写

main.cpp                           ← 阶段一演示：LED 闪烁
pipeline.cpp                       ← 阶段二演示：多线程管道
Makefile                           ← 构建系统（g++ -std=c++11 -pthread）
```

### 对应 Assignment 评分标准的覆盖情况

| 评分项（各 20 分） | 我们的实现 |
|---|---|
| 1. 多线程读 (ThreadedReader) | `ThreadedReader` 类，在 `std::thread` 中循环调用 `FtdiDevice::read()`，数据写入 CircularBuffer |
| 2. 环形缓冲区 (CircularBuffer) | `CircularBuffer` 类，mutex + condition_variable 保证线程安全，支持阻塞的 `write()`/`read()` |
| 3. 多线程写 (ThreadedWriter) | `ThreadedWriter` 类，两个构造函数：一个写 FTDI 设备，一个写文件。在 `std::thread` 中运行 |
| 4. 命令行程序 (pipeline) | `pipeline.cpp`，支持 `--output-file`/`--output-ftdi`/`--freq`/`--duration`/`--bufsize` 参数 |
| 5. 架构文档和 UML | `docs/NEW_FEATURE_PLAN.md` 包含 UML 类图、序列图、数据流图 |


## 三、如何演示（Presentation）

### 准备工作

1. FT245R (UM245R) 模块 + USB 数据线（不能是充电线）
2. 面包板 + LED + 220 欧电阻
3. 接线：DB0 → 电阻 → LED(+) → LED(-) → GND

### 演示流程（建议顺序）

#### 第一步：展示编译（~1 分钟）

```bash
make clean && make pipeline blink_test
```

**要点**：展示所有 7 个 .cpp 文件编译成功，生成 `libioLibrary.a` 静态库，链接出两个可执行文件。

#### 第二步：演示 blink_test — 阶段一成果（~2 分钟）

```bash
./blink_test
```

**展示**：
- LED 以 1Hz 闪烁（每秒亮灭一次，约 10 秒）
- LED 以 2Hz 闪烁（每 0.5 秒亮灭一次，明显更快，约 10 秒）
- 终端输出 `Write cycle 0: 0xFF`（亮）和 `Write cycle 1: 0x00`（灭）交替

**讲解要点**：
> "这是阶段一的成果。ioWrite 类按配置的频率向设备写入 0xFF（全亮）和 0x00（全灭），实现 LED 闪烁。所有 FTDI 调用都封装在 FtdiDevice 类中。"

#### 第三步：演示 pipeline — 阶段二成果（~3 分钟）

```bash
./pipeline --output-file output.bin --freq 5 --duration 5
```

**展示**：
- 终端显示 `[Reader] Thread started` 和 `[Writer] Thread started` — 两个线程启动
- Reader 和 Writer 的 cycle 日志交替出现 — 数据在两个线程间通过 CircularBuffer 传递
- 5 秒后两个线程干净退出，无死锁

**验证输出文件**：
```bash
xxd output.bin | head
```

**讲解要点**：
> "这是多线程管道。Reader 线程从 FTDI 读取引脚状态，放入线程安全的 CircularBuffer。Writer 线程从 CircularBuffer 取出数据，写入文件。两个线程并发运行，通过 mutex 和 condition_variable 同步。"

#### 第四步：展示代码架构（~2 分钟）

打开 `docs/NEW_FEATURE_PLAN.md`，展示：
1. **UML 类图** — 7 个类及其关系（composition / aggregation）
2. **序列图** — 数据从 FTDI → Reader → CircularBuffer → Writer → File 的流向
3. **CircularBuffer 的线程安全设计** — 打开 `CircularBuffer.cpp`，展示 `std::unique_lock` + `wait()` + `notify_one()` 的阻塞同步逻辑

#### 第五步：回答可能的问题

| 可能的问题 | 答案 |
|---|---|
| 为什么用 CircularBuffer 而不是普通 ioBuffer？ | 普通 buffer 没有线程同步，多线程并发读写会数据竞争。CircularBuffer 用 mutex + condition_variable 保证线程安全 |
| 为什么 ThreadedWriter 有两个构造函数？ | 一个接收 `FtdiDevice*`（写入另一个 FTDI 设备），一个接收 `const char*`（写入文件）。满足 assignment 要求的两种输出方式 |
| Reader 和 Writer 速度不一致怎么办？ | CircularBuffer 会自动阻塞：buffer 满时 Reader 等待，buffer 空时 Writer 等待。这是经典的生产者-消费者模式 |
| 如何优雅停止？ | Reader 调用 `stop()` 时设置 `circBuffer->setDone()`，Writer 会把 buffer 里剩余数据写完后退出 |
| 为什么用 C++ 而不是 C？ | C++ 提供 `std::thread`、`std::mutex`、`std::condition_variable`、类封装、RAII 析构——比 C 的 pthread 更安全、更易维护 |
